## 题目一：表格数据的智能查询系统

> **题目属性**：研究探索型，重在思路与探索过程，对最终结果的准确率没有严格下限要求；鼓励在课程结束后继续深入研究。

### 背景

现实中的 Excel 表格往往结构复杂、格式不规范，难以直接用 SQL 查询。SIGMOD 2025 论文 [ST-Raptor: LLM-Powered Semi-Structured Table Question Answering](https://arxiv.org/pdf/2508.18190) 提供了一个中文半结构化表格问答数据集（[SSTQA-zh](https://github.com/OpenDataBox/ST-Raptor/tree/master/data/SSTQA-zh)），可作为本题的测试基准。

**一个可行的设计思路**：没有方案是万能的。原始复杂的 Excel 表格可能部分适合关系型数据库，部分适合 Key-Value 存储，部分需要依赖语义理解。因此，系统设计的重点在于：

1. 对原始表格进行**结构分析与拆分**
2. 根据数据特点，选择合适的**存储方案**（关系型数据库、Key-Value、向量数据库等）
3. 根据用户查询的语义，**路由到合适的查询方式**（SQL、KV 查找、LLM 推理等）

### 具体要求

- 以 SSTQA-zh 数据集中的表格作为输入，实现一个系统，能够：
  1. 将通用表格数据转换并存储到合适的数据结构中
  2. 接收用户的自然语言查询（来自 `test.jsonl`），返回答案
  3. 与标准答案对比，**统计并报告查询正确率**
- 报告中须说明：表格拆分策略、存储方案选择依据、查询路由逻辑

### 评分参考

- 系统设计的合理性与创新性（50%）
- 实现完整度与代码质量（30%）
- 报告写作质量（20%）

## 题目二：轻量级数据库引擎

> **题目属性**：工程实现型，重在代码质量、功能完整性与性能分析。

### 子模块 A：轻量级查询引擎（必做）

**目标**：以 CSV 文件作为外部存储，实现一个支持单表查询的轻量级数据库查询引擎。

**功能要求**：

| 功能     | 说明                                                                        |
| -------- | --------------------------------------------------------------------------- |
| 条件过滤 | 支持 `=`、`>`、`<`、`>=`、`<=`、`!=` 等比较运算符，以及 `AND`/`OR` 逻辑组合 |
| 聚合查询 | 支持 `COUNT`、`SUM`、`AVG`、`MAX`、`MIN`                                    |
| 排序     | 支持 `ORDER BY`（升序/降序）                                                |
| 投影     | 支持指定返回列（`SELECT col1, col2`）                                       |
| CLI 接口 | 必须提供命令行交互界面，用户可输入查询语句并获得结果                        |
| 哈希索引 | 实现哈希索引，并通过实验**对比有无索引的查询耗时**                          |

**技术限制**：

- **禁止**使用现有数据库引擎或查询库（如 Pandas、SQLite、DuckDB、H2 等）
- Python 项目须使用 `uv` 管理；Java 项目须使用 Maven 或 Gradle 管理

**实验要求**：

- 准备至少一个包含 **10 万行以上**数据的 CSV 测试文件
- 对比有无哈希索引时，等值查询（`=`）的响应时间，并在报告中给出分析

### 子模块 B：轻量级列式存储引擎

**目标**：设计并实现列式存储，与行式存储进行性能对比。

**要求**：

- 实现列式存储的读写接口
- 设计实验，对比列式存储与行式存储在**聚合查询**（如 `SUM`、`AVG`）和**全行读取**场景下的查询效率差异
- 在报告中解释性能差异的原因

### 子模块 C：自然语言查询接口

**目标**：实现一个自然语言转 SQL 的查询接口。

**要求**：

- 用户输入自然语言查询（如"查询销售额大于 1000 的所有订单"），系统自动转换为 SQL 并执行
- 须说明使用的 LLM 接口或模型，以及 Prompt 设计思路
- 提供至少 **10 个测试用例**，展示转换结果与执行结果

### 评分参考

- 子模块 A 功能完整性与正确性（40%）
- 哈希索引实现与性能对比实验（20%）
- 代码质量与工程规范（10%）
- 报告写作质量（20%）

## 题目三：培养方案数据库系统

> **题目属性**：工程实现型，重在数据处理能力、数据库设计与查询系统实现。

### 子模块 A：培养方案数据库

**数据来源**：[西南财经大学培养方案](https://jwc.swufe.edu.cn/info/1005/37211.htm)

**要求**：

1. **数据预处理**：从上述网页爬取或下载培养方案数据，进行清洗与结构化处理
   - 考虑到工作量，可选择**部分学院或专业**（须在报告中说明选择范围）
   - 须在报告中说明数据预处理的步骤与遇到的问题

2. **数据库设计**：设计合理的关系型数据库 Schema
   - 须包含 ER 图或表结构说明
   - 须说明主键、外键及约束设计

3. **查询系统**：实现一个查询系统，支持以下典型查询（至少实现其中 4 项）：
   - 查询某专业的必修课列表
   - 查询某门课程的学分、学时信息
   - 查询某专业的总学分要求
   - 查询开设某门课程的所有专业
   - 查询某学院下所有专业的培养方案概览
   - 支持关键词模糊搜索课程名称

4. **用户界面**：提供 CLI 或简单的 Web 界面，方便演示

> [!TIP]
> 不需要浪费时间在UI搭建上，非评分项。建议使用 FastAPI 或 springdoc-openapi 。

### 子模块 B：跨校培养方案对比分析

**数据来源**：[上海财经大学培养方案](https://gongkai.sufe.edu.cn/27/a5/c12262a206757/page.htm)

**要求**：

1. 整合上海财经大学的培养方案数据，与西南财经大学的数据存入同一数据库
2. 实现支持**跨校对比**的 SQL 查询，例如：
   - 对比相同专业（如金融学）在两所学校的课程设置异同
   - 对比两校同一专业的总学分要求
3. 实现**自然语言查询接口**，用户可用中文提问，系统自动转换为 SQL 并返回结果
   - 须提供至少 **5 个跨校对比查询**的测试用例

### 评分参考

- 数据预处理与数据库设计（30%）
- 查询系统功能完整性（30%）
- 代码质量与工程规范（10%）
- 报告写作质量（30%）
